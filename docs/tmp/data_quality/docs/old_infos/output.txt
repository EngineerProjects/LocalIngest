ONGLET 4 — SITES AVEC ANOMALIES (VUE MÉTIER)
──────────────────────────────────────────────────────────────────────────────────────

Objectif :
proposer une vue synthétique, lisible et exploitable par les équipes métier.

Principe :
1 ligne = 1 site
(et non 1 ligne = 1 anomalie)

Cette vue sert à :

- identifier rapidement les sites à corriger ;
- prioriser les traitements ;
- comprendre les actions à mener.

Colonnes du tableau
───────────────────

┌────────────────────────────┬────────────────────────────────────────────────────┐
│ Colonne                    │ Description                                        │
├────────────────────────────┼────────────────────────────────────────────────────┤
│ ID_SITE                    │ Identifiant unique du site                         │
│ NU_CNT                     │ Numéro du contrat associé                          │
│ PAYS_SITE                  │ Pays du site                                       │
│ SUPPORT_LEVEL              │ Niveau de support du pays                          │
│ IS_LOCALIZED               │ TRUE / FALSE                                       │
│ LOCALIZATION_MODE          │ GPS / ADRESSE / AUCUN                              │
│ NB_ANOMALIES               │ Nombre total d’anomalies sur le site               │
│ WORST_SEVERITY             │ Gravité maximale                                   │
│ ANOMALY_CODES              │ Liste des codes anomalies                          │
│ ANOMALY_SUMMARY            │ Résumé lisible des problèmes                       │
│ PRIORITE                   │ HIGH / MEDIUM / LOW                                │
│ HAS_TECHNICAL_ISSUE        │ TRUE / FALSE                                       │
│ TECHNICAL_FIELDS_MISSING   │ Liste des champs techniques manquants              │
│ ACTION_RECOMMANDEE         │ Action principale à effectuer                      │
└────────────────────────────┴────────────────────────────────────────────────────┘



LOGIQUE DE CONSTRUCTION DES COLONNES MÉTIER
──────────────────────────────────────────────────────────────────────────────────────

1. ANOMALY_SUMMARY
------------------
Résumé simplifié des problèmes détectés, destiné à une lecture rapide.

Exemples :
- Site non localisable + incohérence CP / Ville
- GPS hors pays
- Données techniques manquantes
- Adresse incomplète


2. PRIORITE
-----------
Déterminée à partir de la gravité maximale rencontrée :

┌──────────────┬──────────┐
│ Gravité      │ Priorité │
├──────────────┼──────────┤
│ GRAVE        │ HIGH     │
│ LÉGÈRE       │ MEDIUM   │
│ INFO         │ LOW      │
└──────────────┴──────────┘


3. ACTION_RECOMMANDEE
---------------------
Déduite des anomalies principales détectées sur le site.

┌──────────────────────────────┬──────────────────────────────────────────────────────┐
│ Cas                          │ Action recommandée                                   │
├──────────────────────────────┼──────────────────────────────────────────────────────┤
│ L-01                         │ Corriger GPS ou adresse (site non localisable)       │
│ G-05                         │ Vérifier les coordonnées GPS (hors pays)             │
│ G-01 / G-02 / G-03           │ Compléter ou corriger le GPS                         │
│ A-01 / A-02                  │ Compléter l’adresse                                  │
│ A-04                         │ Corriger le code postal                              │
│ A-05                         │ Vérifier la cohérence code postal / ville            │
│ P-07                         │ Renseigner les données techniques                    │
└──────────────────────────────┴──────────────────────────────────────────────────────┘

                                   SYNTHÈSE FINALE
══════════════════════════════════════════════════════════════════════════════════════

Le dispositif de contrôle qualité Proxima repose sur une logique simple, robuste et
orientée métier :

- exploiter en priorité le GPS ;
- utiliser l’adresse en solution de secours ;
- signaler clairement les anomalies ;
- distinguer ce qui bloque réellement de ce qui reste corrigeable ;
- produire un rapport à la fois analytique et opérationnel.

Cette organisation permet d’obtenir un contrôle fiable de la qualité des données de
localisation, tout en facilitant la correction des données par les équipes concernées.

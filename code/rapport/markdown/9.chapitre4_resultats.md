# CHAPITRE 4 — PRÉSENTATION ET ANALYSE DES RÉSULTATS

## 4.1 Architecture finale implémentée

L'architecture médaillon définie en phase de conception a été intégralement déployée. Elle gère deux canaux de distribution distincts, AZ (agents et courtiers) et AZEC (entreprises de construction), avant de les consolider. Les données sont organisées dans Azure Data Lake Storage selon la hiérarchie suivante :

```
{conteneur_datalake}/
├── bronze/
│   ├── {YYYY}/{MM}/    ← données mensuelles (ipf16.csv, polic_cu.csv, capitaux, primes...)
│   └── ref/            ← référentiels stables (ISIC, NAF, FFB, IRD Risk, SIRET)
├── silver/
│   └── {YYYY}/{MM}/    ← résultats intermédiaires par pipeline
└── gold/
    └── {YYYY}/{MM}/    ← cubes finaux consommables par la BI
```

Le partitionnement temporel (`{YYYY}/{MM}/`) permet un traitement incrémental : chaque vision mensuelle est traitée indépendamment, sans recalcul de l'historique. Au total, quarante-trois groupes de fichiers (*file groups*) sont définis dans la configuration du système, couvrant l'ensemble des sources identifiées lors du recensement. Les schémas Spark de chaque fichier source sont formalisés dans des objets `StructType`, garantissant une lecture typée et la détection précoce d'anomalies structurelles dans les données d'entrée.

### 4.1.3 Couche Silver — Transformations métier par pipeline

La couche Silver reçoit les données de Bronze et applique l'ensemble des transformations métier propres à chaque pipeline. Chaque pipeline produit plusieurs tables Silver correspondant aux canaux de distribution :

Pour le **pipeline PTF Mouvements**, la couche Silver produit deux tables distinctes : `mvt_const_ptf_{vision}` pour le canal AZ (agent et courtage) et `azec_ptf_{vision}` pour le canal AZEC. Ces tables contiennent les données de portefeuille filtrées sur le marché Construction, avec les indicateurs de mouvements calculés (AFN, RES, PTF, RPT, RPC), les capitaux extraits (SMP, LCI, PE, RD et dix autres champs), et les expositions calculées (YTD et GLI).

Pour le **pipeline Capitaux**, la couche Silver produit les tables de capitaux AZ et AZEC avec l'indexation FFB appliquée : chacun des quatorze champs de capitaux est calculé à la fois en valeur nominale et en valeur indexée, permettant les analyses actuarielles de l'évolution des garanties dans le temps.

Pour le **pipeline Émissions**, la couche Silver contient les données de primes après application des filtres d'exclusion métier, prêtes pour l'agrégation vers les deux formats de sortie Gold.

### 4.1.4 Couche Gold — Consolidation et enrichissements finaux

Au total, l'architecture produit **sept fichiers Parquet distincts** par vision mensuelle traitée, répartis entre les couches Silver et Gold :

*Tableau 4.1 — Fichiers de sortie produits par vision mensuelle*

| **Couche** | **Pipeline** | **Fichier** | **Contenu** |
|---|---|---|---|
| Silver | PTF Mouvements | `mvt_const_ptf_{vision}.parquet` | Mouvements de portefeuille AZ |
| Silver | PTF Mouvements | `azec_ptf_{vision}.parquet` | Mouvements de portefeuille AZEC |
| Silver | Capitaux | `capitaux_az_{vision}.parquet` | Capitaux AZ avec indexation FFB |
| Silver | Capitaux | `capitaux_azec_{vision}.parquet` | Capitaux AZEC avec indexation FFB |
| Gold | PTF Mouvements | `az_azec_ptf_{vision}.parquet` | Cube consolidé PTF (AZ + AZEC + enrichissements) |
| Gold | Capitaux | `az_azec_capitaux_{vision}.parquet` | Cube consolidé Capitaux (AZ + AZEC) |
| Gold | Émissions | `primes_emises_{vision}.parquet` | Primes par police et par garantie |

Les fichiers Gold constituent les cubes directement exploitables par les outils de Business Intelligence d'Allianz. Ils sont stockés au format Parquet avec optimisation de la compression, et incluent l'ensemble des variables d'enrichissement (ISIC, IRD, clients, produits spéciaux) calculées lors des étapes de consolidation.

## 4.3 Résultats par pipeline

### 4.3.1 Pipeline PTF Mouvements

Le pipeline PTF Mouvements est le plus complexe des trois, impliquant les deux canaux de distribution, quatorze étapes de transformation métier, et des enrichissements multiples. Sur la vision de référence (décembre 2025), il ingrère 70 907 lignes du canal AZ et 16 124 lignes du canal AZEC, pour produire **87 012 lignes consolidées** en sortie Gold. Le fichier `mvt_ptf_202512.parquet` généré pèse environ 23,4 Mo pour un temps d'exécution de l'ordre de huit minutes. L'exécution est sans erreur.

Les quatorze étapes de transformation du processor AZ s'exécutent de manière séquentielle et indépendante, avec des logs de comptage à chaque étape permettant de vérifier la cohérence des volumes intermédiaires. Le processor AZEC applique une logique de calcul des mouvements spécifique à ce canal, notamment pour le calcul du chiffre d'affaires depuis la table `MULPROCU` et la gestion de la migration AZEC vers IMS via la table de référence `ref_mig_azec_vs_ims`.

La consolidation AZ + AZEC génère l'intégralité des soixante-dix colonnes du cube final, incluant toutes les variables d'enrichissement (ISIC, IRD, clients, produits spéciaux). La distribution du portefeuille entre les deux pôles de distribution est conforme aux attentes métier :

*Tableau 4.2 — Distribution du portefeuille par pôle (pipeline PTF Mouvements, vision 202512)*

| **Pôle** | **Canal** | **Polices Python** | **Polices SAS** | **Écart** |
|---|---|---|---|---|
| Pôle 1 (Agent) | AZ | 32 319 | 32 319 | 0,00% |
| Pôle 3 (Courtage) | AZ + AZEC | 54 693 | 54 693 | 0,00% |
| **Total** | | **87 012** | **87 012** | **0,00%** |

### 4.3.2 Pipeline Capitaux

Le pipeline Capitaux traite l'extraction et l'indexation des garanties financières associées aux contrats de construction. Son architecture reprend la même structure AZ/AZEC/Consolidation que le pipeline PTF Mouvements, tout en réutilisant les fonctions d'extraction des capitaux développées pour ce dernier. Sur la vision 202512, le pipeline ingère 242 927 lignes AZ et 9 588 lignes AZEC, pour produire **248 438 lignes en sortie Gold** avec l'indexation FFB intégralement appliquée, en environ cinq minutes.

La particularité de ce pipeline réside dans l'indexation des capitaux selon les indices de la Fédération Française du Bâtiment (FFB). Pour chaque contrat, chacun des quatorze champs de capitaux est calculé à la fois en valeur nominale (valeur contractuelle) et en valeur indexée (ajustée selon le coefficient d'évolution du coût de la construction). La normalisation à 100% des coassurances garantit l'homogénéité du traitement entre polices en assurance directe et polices en coassurance proportionnelle.

### 4.3.3 Pipeline Émissions

Le pipeline Émissions traite les données de primes issues de l'outil de gestion One BI d'Allianz. Il se distingue des deux pipelines précédents par l'absence de consolidation AZ/AZEC et par la production de deux tables de sortie distinctes selon le niveau de granularité requis. Sur la vision 202512, les 1 173 238 lignes brutes sont réduites à 273 796 après filtrage marche Construction (`cd_marche = '6'`) et exclusions métier configurées, soit une réduction de plus de 75%. Le temps d'exécution est d'environ trois minutes. Les primes distinguent l'exercice courant (N) et les rappels des exercices antérieurs (X), conformément à la logique SAS.

## 4.4 Validation de la parité fonctionnelle

### 4.4.1 Méthodologie de comparaison

La validation de la parité fonctionnelle entre les sorties SAS et Python est organisée selon une approche à plusieurs niveaux de granularité, décrite au chapitre 3. Les comparaisons sont effectuées sur la vision de référence (décembre 2025, codée `202512`), période pour laquelle les sorties SAS de production sont disponibles et servent de référence.

### 4.4.2 Résultats de validation structurelle

La validation structurelle vérifie l'existence des fichiers de sortie, la conformité des schémas (colonnes et types) et la cohérence globale des volumes. L'ensemble des tableaux de sortie Python présente les colonnes attendues avec les types corrects, sans valeur nulle inattendue sur les colonnes obligatoires.

Un exemple détaillé de validation de la correction critique `CDPOLE` illustre le niveau de rigueur appliqué :

*Tableau 4.5 — Impact de la correction de l'assignation CDPOLE sur la distribution du portefeuille (vision 202512)*

| **Pôle** | **Avant correction** | **Après correction** | **Référence SAS** | **Statut** |
|---|---|---|---|---|
| Pôle 1 — Agent | 32 319 | 32 319 | 32 319 | Match |
| Pôle 3 — Courtage | 38 569 | **54 693** | 54 693 | Match |
| Sans cdpole (AZEC non assigné) | **16 124** | 0 | 0 | Corrigé |
| **Total** | **87 012** | **87 012** | **87 012** | Match |

Avant correction, les seize mille cent vingt-quatre polices du canal AZEC n'avaient aucune valeur pour la colonne `cdpole` : le processor AZEC Python créait bien les colonnes de résultats mais omettait d'assigner le code pôle, contrairement à SAS qui le fait explicitement dans la macro de consolidation en affectant systématiquement la valeur `"3"` à toutes les lignes AZEC. Après correction, ces lignes sont intégralement rattachées au Pôle 3 (Courtage), conformément à la logique SAS. La sortie finale ne contient donc aucune valeur manquante sur `cdpole`.

### 4.4.3 Résultats de validation fonctionnelle (KPI)

La validation fonctionnelle s'appuie sur le fichier `kpis_results.xlsx` qui centralise les comparaisons SAS vs Python pour chaque indicateur métier, décomposé par code produit (`CDPROD`). Les tests sont conduits sur dix visions mensuelles distinctes.

**Pipeline PTF Mouvements** : sur la vision 202512, la parité est totale, 87 012 polices en SAS, 87 012 en Python, zéro écart sur l'ensemble des KPI métier (NBPTF, NBAFN, NBRES, NOPOL) pour chaque code produit testé. Le nombre de colonnes est également identique, 110 colonnes en SAS, 110 en Python. Les captures d'écran de la feuille KPI PTF du fichier `kpis_results.xlsx` ci-dessous illustrent ce résultat sur plusieurs visions concurrentes.

**Pipelines Capitaux et Émissions** : les grilles de comparaison identiques sont établies pour les indicateurs `VALUE_INSURED`, `RISQUE_DIRECT` (Capitaux) et `Primes_N`, `Primes_X`, `MTCOM_X` (Émissions). Ces comparaisons sont en cours sur les dix visions prévues ; les résultats complets sont consignés dans `kpis_results.xlsx` (voir Annexe A).


### 4.4.4 Statut de la validation complète et point d'attention Émissions

La validation structurelle et les KPI de volume sont intégralement confirmés sur la vision de référence 202512. Pour les pipelines PTF Mouvements et Capitaux, les tables de sortie Python présentent exactement le même nombre de lignes que les tables SAS de référence, les mêmes distributions par code pôle (`cdpole`) et par code produit (`cdprod`), et une structure colonne identique. Cette parité, obtenue sur l'ensemble des indicateurs de volume et de structure, atteste de la rigueur de l'implémentation et de la fidélité de la migration par rapport au comportement SAS original. La validation de parité sur dix visions historiques distinctes, couvrant les sorties Capitaux et Émissions sur des périodes supplémentaires, est en cours et fera l'objet d'un rapport de validation complet dans le fichier `kpis_results.xlsx`.

## 4.5 Analyse de la structure du code Python produit

### 4.5.1 Modularité et réutilisabilité

L'un des critères d'évaluation de la migration, au-delà de la parité fonctionnelle, est la qualité du code produit et sa capacité à être étendu aux autres marchés du programme SAS EXIST. Sur ce point, la structure adoptée présente plusieurs atouts significatifs.

Le **module de lecture** (`src/reader.py`) est entièrement générique : il lit n'importe quel file group défini dans `reading_config.json`, gère les partitionnements mensuels et référentiels, et valide les schémas automatiquement. Il est directement réutilisable sans modification pour les futurs datamarts.

Les **fonctions utilitaires** (`utils/transformations/`) encapsulent les traitements métier les plus récurrents, filtres de marché, calcul des expositions, extraction des capitaux, enrichissement ISIC, sous forme de fonctions paramétrées. La configuration externalisée en JSON permet d'adapter ces fonctions à un autre marché (Auto, Habitation) en modifiant uniquement les fichiers de configuration, sans réécriture de logique.

Le **logger personnalisé** (`utils/logger.py`) offre des méthodes dédiées aux points de contrôle (`step()`, `section()`, `success()`) qui produisent des logs structurés incluant systématiquement les comptages de lignes. Ces logs ont joué un rôle central dans le diagnostic des anomalies lors de la validation.

### 4.5.2 Configuration externalisée

La totalité des paramètres métier susceptibles d'évoluer est externalisée en dehors du code applicatif : définition des 43 *file groups* avec schémas et chemins (`reading_config.json`), constantes métier (`constants.py`), filtres et mappings de colonnes (`transformations/*.json`), et schémas Spark formalisés (`schemas.py`). Le détail de l'organisation des fichiers de configuration est documenté dans le dépôt GitHub (voir Annexe A).

Cette architecture de configuration garantit qu'une modification des règles métier (ajout d'un intermédiaire exclu, changement d'un seuil de filtrage) n'implique aucune modification du code Python et peut être réalisée par un analyste dont la maîtrise du code ne serait pas nécessairement approfondie.

### 4.5.3 Comparaison qualitative SAS vs Python

Au-delà de la parité fonctionnelle des résultats, la migration apporte des améliorations qualitatives structurelles au regard du système source :

*Tableau 4.8 — Comparaison qualitative SAS vs Python sur les critères de maintenabilité*

| **Critère** | **SAS (avant migration)** | **Python/PySpark (après migration)** |
|---|---|---|
| Versionning du code | Absence de Git, versioning manuel | Git intégré, historique complet des modifications |
| Documentation | Absente, logique dans le code uniquement | Double documentation (fonctionnelle + technique) |
| Modularité | Macros monolithiques (~500 à 1 000 lignes) | Modules de ~150 lignes, responsabilités isolées |
| Configuration | Paramètres hardcodés dans le code | Externalisée en JSON/YAML, modifiable sans code |
| Traçabilité | Logs SAS limités | Logs structurés avec comptages à chaque étape |
| Testabilité | Difficile (environnement SAS propriétaire) | Tests unitaires possibles par module |
| Scalabilité | Serveur dédié SAS, non élastique | Cluster Spark élastique sur Azure Databricks |

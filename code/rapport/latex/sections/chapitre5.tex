\chapter{Discussion et perspectives} 

\section{Analyse critique du projet}

\subsection*{Ce qui a bien fonctionné}

Le résultat le plus concret est simple : les trois pipelines tournent, les sorties PTF et Capitaux sont identiques au SAS, et une documentation complète existe désormais. Aucun de ces trois points n’était acquis au départ.

Le choix d’\textbf{analyser avant de coder} a été déterminant. Les deux semaines initiales consacrées à la lecture du code SAS ont évité de nombreuses erreurs d’interprétation qui auraient été difficiles à rattraper en fin de projet. Sur une migration sans documentation initiale, c’était indispensable.

L’\textbf{architecture médaillon} a aussi été un vrai avantage pendant la validation. Comme chaque couche est persistée, il est possible de vérifier les données étape par étape et d’identifier précisément l’origine d’une divergence. Sans cette séparation, déboguer la correction \texttt{cdpole} aurait été beaucoup plus complexe.

Enfin, la \textbf{configuration externalisée} s’est révélée très pratique : corriger un filtre ou ajouter un code produit ne nécessite pas de modifier le code Python, seulement un fichier JSON. C’est l’opposé de SAS, où les paramètres étaient éparpillés dans le code.

\subsection*{Ce qui a été difficile}

La difficulté principale a été \textbf{l’absence totale de documentation SAS}. Toute la logique se trouvait dans des macros imbriquées, avec des variables peu explicites. Cela a ralenti le démarrage et laisse toujours un risque résiduel : certaines règles implicites peuvent exister sans être visibles dans le code.

L’\textbf{accès aux données de test} a également été un défi. Les règles de sécurité Azure d’Allianz sont strictes et l’obtention d’extraits réels pouvait prendre du temps. Plusieurs transformations ont dû être codées sans données réelles, puis ajustées une fois l’accès obtenu.

\section{Positionnement par rapport aux pratiques de l'industrie}

La solution développée est cohérente avec les standards actuels du data engineering. L’architecture médaillon est la recommandation de référence pour les architectures \textit{lakehouse}, et le choix d’Azure Databricks correspond à la stratégie cloud d’Allianz.

Par rapport à d’autres approches observées dans l’assurance, notamment celles basées sur Pandas qui atteignent vite leurs limites, PySpark permet de traiter des volumes bien supérieurs. La structure modulaire du code, conforme aux conventions Python, marque aussi une vraie rupture avec le SAS d’origine.

\section{Ce que ça apporte à Allianz France}

Trois bénéfices concrets ressortent :

\begin{itemize}
    \item \textbf{Réduction des coûts} : la migration permet à terme de réduire les licences SAS.
    \item \textbf{Scalabilité} : Spark s’adapte automatiquement aux volumes croissants, ce qui n’est pas le cas de l’environnement SAS actuel.
    \item \textbf{Capitalisation documentaire} : avant le projet, la connaissance du datamart Construction était uniquement dans le code SAS. Elle est maintenant formalisée dans plusieurs documents utilisables par toute l’équipe.
\end{itemize}

Deux points de vigilance se dégagent également : la période de coexistence SAS/Python, qui a un coût opérationnel, et la montée en compétences PySpark des équipes, qui doit être accompagnée.

\section{Perspectives}

\textbf{Court terme} : finaliser la validation des dix visions historiques pour les pipelines Capitaux et Émissions. Le pipeline PTF Mouvements est déjà stabilisé.

\textbf{Moyen terme} : réutiliser le framework développé (module de lecture, logger, configuration JSON, structure de projet) pour d'autres marchés ABR tels que l'Auto et l'Habitation. Les prochaines migrations pourront se concentrer sur les règles métier spécifiques plutôt que sur les briques techniques.

\textbf{Long terme} : lorsque tous les marchés auront été migrés, l’infrastructure SAS pourra être désactivée. Une évolution naturelle sera l’utilisation avancée de Delta Lake (ACID, time travel) pour les marchés futurs.
